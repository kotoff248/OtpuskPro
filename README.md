# Kabinet.pro

Kabinet.pro - веб-приложение для планирования отпусков и согласования заявок
сотрудников. Проект выполнен как практическая часть магистерской работы.

Основная идея проекта - показать, как можно уйти от ручного ведения графика
отпусков в таблицах. В системе учитываются сотрудники, отделы, руководители,
остатки оплачиваемого отпуска, пожелания работников, пересечения по датам и
кадровая нагрузка подразделений.

Отдельная часть работы - нейромодуль поддержки принятия решений. Он не заменяет
правила предметной области, а помогает выбрать более подходящие варианты среди
тех периодов, которые уже прошли обязательные проверки.

## Основной сценарий

1. HR запускает планирование отпусков на год.
2. Сотрудники заполняют пожелания по основному и запасному периоду.
3. HR формирует черновик графика.
4. Система проверяет доступные дни, пересечения, праздники и риски по отделам.
5. Для незакрытых дней можно использовать действие `Добрать незакрытые дни`.
6. Руководители отделов проверяют свои подразделения и утверждают или возвращают
   график на доработку.
7. После этого график уходит на финальное утверждение.
8. В течение года сотрудники могут создавать заявки и запросы переноса.

Такой сценарий нужен не только для красивого календаря, но и для проверки
управленческой логики: кто имеет право согласовать заявку, как не допустить
самосогласование, что делать с остатками отпуска и как объяснить выбранные
периоды.

## Что реализовано

- авторизация и разные роли пользователей;
- личный кабинет сотрудника;
- справочник сотрудников и отделов;
- календарь отпусков;
- заявки на оплачиваемый, неоплачиваемый и учебный отпуск;
- переносы утвержденных периодов графика;
- расчет оплачиваемого остатка по рабочим годам;
- учет российских праздничных дней при расчете оплачиваемых дней;
- сбор пожеланий сотрудников перед планированием;
- черновик годового графика отпусков;
- автоматическое добирание незакрытых дней;
- проверка рисков по отделам и кадровому покрытию;
- согласование графика руководителями отделов;
- финальное согласование графика;
- уведомления и доска заявок;
- аналитическая страница;
- обучение и использование JSON-артефактов моделей для ранжирования вариантов.

## Роли

В проекте используются следующие роли:

- `employee` - обычный сотрудник;
- `hr` - сотрудник HR;
- `department_head` - руководитель отдела;
- `enterprise_head` - руководитель предприятия;
- `authorized_person` - уполномоченное лицо для отдельных согласований.

Маршрут согласования зависит от роли автора заявки. Например, обычная заявка
сотрудника уходит руководителю отдела, а заявка руководителя или HR - на уровень
руководителя предприятия. Пользователь не может согласовать собственную заявку.

## Нейромодуль

В проекте нейромодуль используется внутри формирования черновика графика
отпусков. Сначала система применяет жесткие правила:

- хватает ли у сотрудника оплачиваемых дней;
- не нарушается ли минимальная непрерывная часть отпуска;
- нет ли недопустимых пересечений;
- сохраняется ли кадровое покрытие отдела;
- попадает ли период в допустимые границы планирования.

Только после этого допустимые варианты оцениваются моделью. В проекте есть два
уровня оценки:

- candidate scorer - оценивает отдельный период отпуска;
- package ranker - оценивает набор периодов сотрудника за год.

PyTorch используется для обучения. В обычной работе Django-приложение читает
сохраненные JSON-артефакты моделей, поэтому страницы не зависят от PyTorch во
время просмотра.

## Структура проекта

```text
apps/
  accounts/    вход, выход и проверка ролей
  core/        уведомления, общие модели и demo-команды
  employees/   сотрудники, отделы и кадровое покрытие
  leave/       отпуска, графики, заявки, согласования, аналитика и ML
config/        настройки Django и корневые маршруты
scripts/       вспомогательные скрипты для локального запуска
static/        CSS и JavaScript
templates/     HTML-шаблоны
```

Основная бизнес-логика находится в `apps/leave/services`. Логика черновика
графика вынесена в `apps/leave/services/schedule_drafts`, а код обучения и
скоринга моделей - в `apps/leave/ml`.

## Основные страницы

- `/` - вход в систему;
- `/main/` - личный кабинет;
- `/employees/` - сотрудники;
- `/departments/` - отделы;
- `/staffing/` - кадровое покрытие;
- `/calendar/` - календарь отпусков;
- `/calendar/planning/` - планирование графика;
- `/applications/` - заявки и переносы;
- `/analytics/` - аналитика;
- `/notifications/` - уведомления.

## Запуск локально

Создать файл `.env` по примеру `.env.example` и заполнить параметры PostgreSQL:

```env
DB_NAME=your_database_name
DB_USER=your_database_user
DB_PASSWORD=your_database_password
DB_HOST=localhost
DB_PORT=5432
```

Установить зависимости:

```powershell
.\.venv\Scripts\python.exe -m pip install -r requirements.txt
```

Применить миграции:

```powershell
.\.venv\Scripts\python.exe manage.py migrate
```

Запустить сервер:

```powershell
.\scripts\django_server.ps1 -Action restart -Port 8001 -ReadyTimeoutSeconds 10
```

После запуска:

```text
http://127.0.0.1:8001/
```

Остановить сервер:

```powershell
.\scripts\django_server.ps1 -Action stop -Port 8001
```

## Демо-данные

Для локальной демонстрации можно заполнить базу тестовыми данными:

```powershell
.\.venv\Scripts\python.exe manage.py seed_vacation_requests --confirm-reset --seed-value 42
```

Команда пересоздает демонстрационные данные: сотрудников, отделы, графики,
заявки, переносы, уведомления и историю для обучения моделей.

Быстрый вариант:

```powershell
.\.venv\Scripts\python.exe manage.py seed_vacation_requests --confirm-reset --fast
```

## Обучение моделей

Обучение модели отдельных периодов:

```powershell
.\.venv\Scripts\python.exe manage.py train_vacation_candidate_model
```

Обучение модели годовых пакетов:

```powershell
.\.venv\Scripts\python.exe manage.py train_vacation_package_model
```

Версию модели можно выбрать через `.env`:

```env
VACATION_CANDIDATE_SCORER_VERSION=vacation-candidate-mlp-v2
```


Тесты основных приложений:

```powershell
.\.venv\Scripts\python.exe manage.py test apps.accounts apps.employees apps.leave apps.core --keepdb
```

## Статус

Проект находится в демонстрационном состоянии для защиты. Основной рабочий
сценарий построен вокруг планирования графика отпусков, объяснимого подбора
периодов и маршрута согласования.
